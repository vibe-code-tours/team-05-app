# ADR-002: Backend Framework

- **Status:** Accepted
- **Date:** 2026-07-10

## Context

CrossMart's backend must handle a multi-tenant marketplace with:

- Multiple user roles (buyer, seller, admin) with distinct permissions
- High-throughput operations: product listings, order processing, real-time cargo tracking
- Background job processing (email notifications, image resizing, cargo status sync)
- WebSocket support for real-time features (chat, tracking updates)
- Clean separation of concerns as the codebase grows with multiple developers
- Integration with PostgreSQL (via Prisma), Redis (caching/queues), Cloudflare R2 (file storage)
- API consumed by Next.js frontend (SSR data fetching) and potentially mobile apps

The backend needs to be maintainable, well-structured, and support the team's TypeScript-first approach without excessive boilerplate.

## Decision

We chose **NestJS** as the backend framework over the following alternatives:

| Alternative | Why Not |
|-------------|---------|
| **Express.js** | Unopinionated architecture leads to inconsistent patterns as the team grows; no built-in DI, validation, or module system; requires assembling many third-party packages for structure |
| **Fastify** | High performance but similar to Express in lacking architectural conventions; smaller middleware ecosystem; less enterprise-oriented documentation |
| **AdonisJS** | Full-stack framework with its own ORM (Lucid); would conflict with Prisma preference; smaller community than NestJS; opinionated in ways that don't align with our stack |

### Key Reasons

1. **Architectural consistency:** NestJS enforces a modular, decorator-based architecture (modules, controllers, services, guards, pipes) that scales with team size.
2. **TypeScript-native:** Built from the ground up for TypeScript; decorators, dependency injection, and type inference are first-class citizens.
3. **Dependency Injection:** Built-in DI container manages service lifecycles, making testing and module boundaries clean without manual wiring.
4. **Prisma integration:** `@nestjs/prisma` module provides seamless Prisma client injection and lifecycle management.
5. **Microservices ready:** If the platform scales, NestJS supports HTTP, TCP, Redis, and gRPC transports for splitting into microservices without framework migration.
6. **Enterprise patterns:** Guards (authorization), interceptors (logging/transformation), pipes (validation), and filters (exception handling) provide a consistent way to handle cross-cutting concerns.
7. **BullMQ integration:** `@nestjs/bullmq` module for background job processing integrates cleanly with NestJS's module system and Redis infrastructure.

## Consequences

### Positive

- **Consistent architecture:** New developers follow established patterns (module -> controller -> service -> repository) without debating folder structure.
- **Type safety end-to-end:** Prisma schemas, DTOs, and service interfaces create strong type contracts across the API layer.
- **Testable by default:** DI and module isolation make unit and integration testing straightforward; mocking services is natural with NestJS's testing utilities.
- **Scalability path:** If CrossMart grows, individual modules (orders, products, cargo) can be extracted into separate microservices with minimal refactoring.
- **Rich ecosystem:** Official packages for Prisma, BullMQ, Redis, Passport (auth), Swagger (API docs), and GraphQL reduce integration effort.

### Negative

- **Learning curve:** Decorator-based patterns and NestJS's specific conventions (modules, providers, lifecycle hooks) require onboarding time for developers familiar with Express-style middleware.
- **Boilerplate:** Setting up modules, DTOs with class-validator, and Swagger documentation involves more initial code than a minimal Express server.
- **Opinionation overhead:** For simple endpoints, NestJS's structure adds overhead; the framework shines with complex domains but can feel heavy for CRUD-only services.

### Neutral

- NestJS is backed by Nx and has strong community support with regular releases and active maintenance.
- The framework's Express/FASTIFY adapter layer allows using existing Express/Fastify middleware when needed, preserving ecosystem access.
- Background job processing via BullMQ is decoupled from the HTTP layer, allowing independent scaling of workers.
